# Legacy Gate 5 Annex 2 XML builder — review

Plain-English review of `engine/gate_5_delivery/xml_builder_annex2.py` (the old
Annex 2 / auth.099 XML builder) against the real XSD
(`DRAFT1auth.099.001.04_1.3.0.xsd`), the sample message, and the new
`config/delivery/annex2_field_xsd_path_map.yaml`.

**Bottom line:** the legacy builder is **path-driven from the ESMA mapping
workbook** — and that workbook is a genuinely valuable, XSD-validatable crosswalk
(it resolved ~89 of our previously-unmapped fields, see Task 3 below). But the
builder also performs **unsafe silent fill** (ND5 default injection and value
coercion) and assumes a **wide one-row-per-loan** shape with **singleton, non-
repeating collateral**. It must not be wired into the production path as-is.

> No production XML is generated by this review. Production gates remain
> `xml_generation_allowed=false`, `ready_for_xml_delivery=false`.

## 1. What input shape does it expect?

A **wide CSV** (`pd.read_csv(--input)`), one **column per ESMA code** and one
**row per loan**, plus a mapping **workbook** (`.xlsx`, sheet
`DRAFT1auth.099.001.04`, header row 3) with columns `RTS Field code`, `XML TAG`,
`PATH`, `MULTIPLICITY`, `Template`, `Performing/Non Performing`, and a code-order
YAML. The workbook `PATH` already contains the **full XML path including the leaf
tag** — that is the builder's source of truth for node creation.

## 2. Wide rows or long target-frame rows?

**Wide rows.** It iterates `df.iterrows()` and reads `row[code]` / `df[code]`.
This is the opposite of the Delivery/XML Agent's **long** target frame (one row
per loan×field). So it is not directly compatible with the current pipeline.

## 3. RREL and RREC — flat siblings or nested?

**Neither hardcoded — it builds whatever the workbook PATH says.** Because the
workbook paths place RREC under `.../PrfrmgLn/Coll/...`, the builder nests
collateral correctly *for valid paths*. The risk is in multi-code-cell
pollution (see Q12 / Task 2): a shared `RTS Field code` cell can map a collateral
code (e.g. RREC1, RREC2) to a non-collateral element (`ScrtstnIdr`,
`NewUndrlygXpsrIdr`). The builder would follow that polluted path literally.

## 4. Collateral nested under loan?

**Yes**, via the workbook path (`UndrlygXpsrRcrd/UndrlygXpsrData/ResdtlRealEsttLn/
PrfrmgLn/Coll`). The builder anchors records on `UndrlygXpsrRcrd` and builds
`Coll` as part of the path chain.

## 5. Repeating collateral?

**Effectively no.** `Coll` is created via `_get_or_create_singleton`, and one CSV
row produces one `UndrlygXpsrRcrd`. Only *leaf* values can repeat (pipe-separated
via `_split_multi_value` when `maxOccurs > 1`). Multiple `Coll` records per loan
(the XSD allows `Coll` `maxOccurs="unbounded"`) is **not** supported by the
wide-row model.

## 6. Does it insert default values?

**Yes — unsafe.** `_ensure_scndry_oblgr_incm_defaults` forces a `ScndryOblgrIncm`
branch, and `_ensure_hstrcl_colltn_nd_defaults` injects 36 months × 4 blocks of
defaults. `_coerce_record_value_for_branch` **fabricates** a value (`RREL12` →
`"2026"` when the input is not a year). These are silent fills the Delivery/XML
Agent explicitly forbids.

## 7. Does it inject ND values?

**Yes — unsafe.** The default-injection helpers write `NoData = ND5` into
`ScndryOblgrIncm` (income value + verification) and into the historical-collection
months whenever the source is silent. ND is injected to satisfy mandatory XSD
branches, not because the data is genuinely No-Data.

## 8. Does it understand NoDataOptn wrappers?

**Yes.** `select_specs_for_value` routes `ND[1-5]` values to the `NoData` branch
under `.../NoDataOptn/`, and non-ND values to the value branch — and it never
selects a NoData spec for a real value. This NoDataOptn-routing **concept is
reusable** (the new path map records `value_or_nodata` + an `nd_wrapper_path`).

## 9. Does it follow XSD sequence ordering?

**Approximately.** `build_order_index` derives child order from the **workbook
PATH order** (first-seen per parent), and `_append_ordered_new` inserts in that
order. This is a proxy for the XSD `xs:sequence` and is correct only insofar as
the workbook row order matches the schema. It is **not** read from the XSD
sequence directly, so ordering is not guaranteed authoritative.

## 10. Does it produce production-XSD-valid XML?

**Only by injecting defaults.** It has an optional `--xsd` step that validates
with `lxml` and exits non-zero on failure. To *pass* that validation for sparse
inputs it relies on the ND5 default/`ScndryOblgrIncm`/historical injections —
i.e. it can produce XSD-valid XML, but at the cost of **fabricated / silently
filled data**, which is not regulatorily acceptable. Without the injections,
sparse real data would fail validation.

## 11. Which parts are reusable?

- The **workbook PATH crosswalk** (`load_mapping_specs`) — the single most
  valuable asset; full XSD paths per RTS code, re-validatable against the XSD.
- **NoDataOptn routing** (`select_specs_for_value`) — value vs `NoDataOptn/NoData`.
- **Path tree construction** (`get_or_create_path_singleton`,
  `_append_ordered_new`) and **`Amt` `Ccy` attribute** handling.
- Multiplicity parsing (`_parse_multiplicity`) and code-order loading.

## 12. Which parts are unsafe?

- **ND5 default injection** (`_ensure_scndry_oblgr_incm_defaults`,
  `_ensure_hstrcl_colltn_nd_defaults`, `_ensure_nprf_nonprfrmgdata_defaults`).
- **Value fabrication** (`_coerce_record_value_for_branch`, RREL12 → `2026`).
- **Wide one-row-per-loan** input shape and **singleton, non-repeating Coll**.
- **Workbook-order (not XSD-order)** sequencing.
- **Multi-code-cell pollution**: a shared `RTS Field code` cell maps unrelated
  codes (e.g. RREC1/RREC2) to the wrong element; the builder follows it blindly.
- Header cross-row variance handling raises rather than carrying a blocker.

## Verdict

- **Reuse (as evidence/logic, re-validated against the XSD):** the workbook path
  crosswalk, NoDataOptn routing, ordered tree construction, amount-currency
  handling.
- **Retire (never port into production):** all default/ND5 injection and value
  coercion, the wide-row + singleton-Coll model, and any reliance on workbook
  ordering or multi-code-cell paths without XSD validation.

See `output/config_review/legacy_gate5_vs_xsd_path_map.csv` for the per-code
comparison and `docs/annex2_field_xsd_path_mapping.md` (Legacy Gate 5 comparison)
for what was upgraded and what still blocks production.

## Legacy Gate 5: map reused, runtime retired

**Why the workbook PATH column is valuable.** The legacy builder never hard-codes
paths — it reads them from the ESMA mapping workbook, whose `PATH` column is the
full XSD path per RTS code. That crosswalk is authoritative and re-validatable:
every RREL/RREC workbook path validates against the vendored XSD tree. It is the
single best source for *where each field goes*, so we keep it.

**Why the old Gate 5 runtime is unsafe.** The builder's *behaviour* fabricates and
silently fills: it injects `ND5` defaults (`ScndryOblgrIncm`, 36-month historical
collection), coerces values (`RREL12 → "2026"`), assumes a wide one-row-per-loan
CSV, builds a singleton non-repeating `Coll`, orders by workbook (not XSD)
sequence, and follows multi-code-cell paths literally. None of that may enter the
production path.

**What `workbook_xsd_validated` means.** A path that comes from the ESMA workbook
**and** re-validates against the XSD tree — strong corroboration, but **not**
sample-confirmed and **not** production-eligible until the path-map review policy
formally accepts it. It may be implemented in the future builder only *behind a
structure gate*. See `docs/annex2_path_map_promotion_policy.md`.

**Why production XML remains gated.** Only 11 fields are `confirmed_by_xsd_sample`;
89 are `workbook_xsd_validated` (pending acceptance); 7 are manual/unresolved/
conflict. The XSD is still the DRAFT, and — separately — **data readiness** is not
certified for any field. `production_ready` is `false` for all 107 fields.

**What needs manual acceptance before paths drive production XML.** Final non-DRAFT
XSD; human sign-off on each code↔element crosswalk; cardinality/`minOccurs`
reconciliation (esp. repeating `Coll`); value↔`NoDataOptn` confirmation; and a
full-builder XSD validation pass. Until then: **map reused, runtime retired.**
