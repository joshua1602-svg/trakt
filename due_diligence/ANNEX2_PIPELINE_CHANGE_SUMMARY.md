# Annex 2 Pipeline Change Summary (auth.099)

**Date:** 2026-03-29  
**Scope:** Gate 4 (regime projection) + Gate 5 (Annex2 XML builder) only.

---

## 1) What changed in the Annex 2 pipeline

## Gate 5 (`xml_builder_annex2.py`)

- Added a dedicated Annex 2 XML builder (`engine/gate_5_delivery/xml_builder_annex2.py`) to avoid reusing Annex 12-specific XML assumptions.
- Implemented workbook-driven path semantics for Annex 2:
  - `PATH` is treated as full XML path (including leaf).
  - `XML TAG` is used for branch selection/validation hints.
- Implemented deterministic row-level record emission:
  - A **fresh `UndrlygXpsrRcrd` branch is created per CSV row**.
  - Header/singleton nodes are handled separately from record-level nodes.
- Added deterministic branch handling:
  - ND detection is strict (`ND1..ND5`).
  - Non-ND values do not route to NoData/Sgn branches.
  - ND routes to NoData branches (NoDataOptn paths) deterministically.
- Added multiplicity-aware emission (`|`-delimited values can emit repeats when allowed).
- Added Annex 2 amount attribute support (`Ccy` on `Amt` leaves).
- Added optional XSD validation with fail-fast behavior.

## Orchestrator wiring (`trakt_run.py`)

- Added Annex 2 routing so `--regime ESMA_Annex2` invokes `xml_builder_annex2.py` in Gate 5.
- Added Annex 2 CLI parameters:
  - `--annex2-mapping-workbook`
  - `--annex2-mapping-sheet`
  - `--annex2-xsd`
  - `--annex2-performance-mode`
- Added `--currency` passthrough to Annex 2 builder for amount attributes.

## Gate 4 (`regime_projector.py`)

- Added Annex 2-only post-projection guard layer (after ESMA rename):
  - Header constants support from config:
    - `regime_overrides.ESMA_Annex2.header_constants.RREL1`
    - `regime_overrides.ESMA_Annex2.header_constants.RREL6`
    - legacy fallback: `defaults.securitisation_identifier -> RREL1`
  - Deterministic backfills:
    - `RREL3 <- RREL2` (when missing)
    - `RREC2 <- RREL3` (when missing)
    - `RREL5 <- RREL4` (when missing)
  - Early fail checks:
    - Header-level fields (`RREL1`, `RREL6`) must be non-blank constants across rows.
    - Mandatory identity fields (`RREL3`, `RREL5`, `RREC2`) must be populated.

## 3) Synthetic data points supplemented to close gaps for XSD validation

Synthetic supplementation was used only for test validation and diagnostics:

1. **Header-level securitisation ID constant**
   - Added test override for `RREL1` via temp config:
     - `regime_overrides.ESMA_Annex2.header_constants.RREL1 = "ERE_122025_SEC"`
   - Purpose: enforce pool-level header semantics and remove row-varying IDs.

2. **Header-level cut-off date constant**
   - Added test override:
     - `regime_overrides.ESMA_Annex2.header_constants.RREL6 = "2025-10-31"`
   - Purpose: make header-level date deterministic for report-level XML context.

3. **Record-level deterministic identity backfills**
   - Used in Gate 4 shaping logic (not manual CSV edits):
     - `RREL3 <- RREL2`
     - `RREC2 <- RREL3`
     - `RREL5 <- RREL4`
   - Purpose: populate mandatory Annex 2 identity fields where source data is partially present.

4. **ND branch test points**
   - Included ND values (e.g., `ND5`) in synthetic projected cases to verify NoData branch selection and XSD compliance paths.

---

## 4) Observed data insufficiency in user sample (not Gate 5 regression)

From the user-provided raw sample:

- `RREL1` source was row-varying without a configured report-level constant.
- `RREL5` (`new_obligor_identifier`) and `RREL4` (`original_obligor_identifier`) were both blank, so no deterministic backfill path remained.

Result:

- Gate 4 now fails early with precise Annex 2 diagnostics for these conditions.
- This is intentional fail-fast control behavior and prevents late XML/XSD-stage failures.

---

## 5) Net effect

- Annex 2 pipeline is now stricter and more deterministic:
  - structural XML correctness is handled in a dedicated Annex 2 builder,
  - and projection/data-shaping issues are surfaced earlier in Gate 4 with actionable messages.
